# Котедж Куземський — CLAUDE.md

## Про проєкт

Це робочий простір для управління ремонтом котеджу Ігоря та Діани Куземських.
Площа: 101.34 м², 2 поверхи + горище. Дедлайн: вересень 2026.

## Робочі файли

Завжди працюй лише з цими файлами. Вихідні матеріали (HTML, PDF) лежать у `source/` — це архів.

| Файл | Призначення |
|------|-------------|
| `STATUS.md` | Поточний стан, термінові дії, контакти, хронологія |
| `TASKS.md` | Трекер задач по 7 етапах будівництва |
| `SPECS.md` | Всі погоджені технічні рішення (електрика, сантехніка, опалення) |
| `ROOMS.md` | Вимоги та побажання по кожній кімнаті |
| `BUDGET.md` | Трекер бюджету і витрат |
| `MYKOLA.md` | Бриф для прораба: поточні роботи, наступні кроки, погоджені рішення |
| `STAGES.md` | Покроковий план 7 етапів: стан, питання для Миколи, відкриті рішення |

## Як поводитися

- Мова спілкування: **українська**
- **Ніколи не вигадуй розміри, відстані, площі чи кількості** — якщо конкретне значення не зазначено в робочих файлах або повідомленні Ігоря, явно запитай або скажи «не вказано»
- Коли Ігор повідомляє про завершену роботу → оновлюй чекбокси в `TASKS.md` та статус в `STATUS.md`
- Коли Ігор повідомляє про витрату → оновлюй `BUDGET.md`
- Коли приймається нове технічне рішення → додавай в `SPECS.md` або `ROOMS.md` **і в `MYKOLA.md`** (секція «Погоджені технічні рішення»)
- Коли приймається рішення з таблиці «Відкриті рішення» в `STAGES.md` → оновлюй статус у тій таблиці та знімай блокер з відповідного етапу
- Коли завершується етап або починається новий → оновлюй статус у `STAGES.md` (загальна схема та секція відповідного етапу)
- Перед будь-якою відповіддю — читай відповідний файл, щоб бути в актуальному контексті; якщо питання стосується поточних робіт або інструкцій для прораба — читай також `MYKOLA.md`; якщо питання стосується планування або послідовності кроків — читай також `STAGES.md`
- Не дублюй інформацію між файлами без потреби
- Дати вказуй у форматі ДД.ММ.РРРР

## Стиль написання в MYKOLA.md

Всі секції «Погоджені технічні рішення» пишуться в єдиному стилі:

- **Тільки плоский список** — жодних вкладених підпунктів
- **Формат рядка:** `- Параметр: **значення** (примітка)` — для специфікацій
- **Формат рядка:** `- **Ключове правило** пояснення` — для обмежень і вимог
- **Bold** — лише для ключових значень, розмірів і правил; не для всього тексту
- Em dash (—) для пояснень і уточнень
- ⚠️ — тільки для критичних попереджень
- **Без абзаців, без підзаголовків усередині секції, без emoji ✅/❌**

## Відкриті рішення — де вести

- Єдине місце: **`STAGES.md`**, таблиця «Відкриті рішення, що блокують прогрес»
- Якщо рішення треба з'ясувати у прораба → дублюй також у **`MYKOLA.md`** секція «Відкриті питання»
- **`STATUS.md`** і **`CLAUDE.md`** власних списків не ведуть
- Нове відкрите питання → лише `STAGES.md` (+ `MYKOLA.md` якщо для прораба)
- Рішення прийнято → оновлюй статус у таблиці `STAGES.md` і видаляй з `MYKOLA.md`
